Case management system — from form submission to decision

GovTech Barbados Forms app × Case Management System (CMS). How an application flows today: a citizen submits a form on the Forms app, the submission travels to the CMS with a program code, an MDA team reviews it in a team-scoped queue, and the applicant can — in principle — track it on a status page. Mapped from the CMS sandbox demo (Ministry of Youth science summer camp example), including the handoff gaps and open design questions surfaced in the walkthrough.

The online form

A Forms app service form (e.g. Science Summer Camp 2026), with document uploads supported in the main form.

Works today

Submission email — with the wrong ID

The only email the applicant ever gets: “your form has been submitted”, from the Forms app, quoting an ID number that is different from the case management ID (shorter, for technical reasons). A fix is under way.

Gap

Track-your-application page

A public status page exists at a URL built by convention (…/track/application/{id}) — but the link is never given out. There is no entryway for applicants to reach it.

Gap

Silence

Nothing reaches the applicant while the case sits in review. No emails go out from the CMS to users at all today.

Gap

“Please tell us more” email

The officer's typed question is emailed to the applicant with a link to a small response form. Text only — no document upload on this loop yet.

Works today

Status page copy at decision

Approved: the page just changes — no email. Rejected: “Your application was not successful” with no phone number, no email, no route to contact the team.

Gap

BackstageMDA staff in the CMS

Program is set up and owned by a team

A program is basically identical with a form (kept separate so one program could span multiple forms or seasons). The owning team sees its applications — and only its own.

StaffWorks today

Submission lands in the queue

The CMS reads the program code, files the submission under the right program, and shows it to the associated team in reverse chronological order.

StaffWorks today

Nobody is told a case arrived

No staff-side notifications, and no concept of assignment yet — “is it you? are you assigned to the case?” is still an open design area.

StaffOpen question

Officer reads the case

Case detail shows applicant details plus everything that came in from the form — currently jumbled together and hard to read. Display improvements planned. A “view applicant page” button shows exactly what the citizen sees.

StaffGap

Officer asks for more information

Types a question into the request-more-information flow; the reply comes back onto the case page and timeline. Text only for now — document upload here would be a small addition, but nobody has asked yet.

StaffWorks today

Officer changes the status

Received → accepted, rejected or archived. Real programs may decide in one batch (summer camp waits for all applicants, then allocates and approves in one go) — the workflow behind decisions isn't understood yet.

Archive, export, administer

Approved cases can be archived to clear the main view. From the program page, staff export all applications as CSV. Super admins manage users, create teams and segment work between them.

Support processesSystems & enablers

Forms platform

Hosts the service forms, handles submission and the initial confirmation email, supports document upload in the main form.

Works today

API integration + program codes

The Forms app posts submissions to the CMS with a program code attached (the form code, e.g. science-2026); API keys for the integration are configured in the CMS admin.

Works today

Tracking URL by convention

The status page address is constructed from the submission ID, so either system could share it — once someone does.

Open question

Magic-link sign-in & team access control

Staff sign in by emailed link — no passwords, no single sign-on (something better is wanted). Teams keep one program's cases invisible to another team in the same MDA.

Works today

Email service — capable, underused

The CMS can send email (it already does for sign-in and info requests); adding status-change emails is described as very straightforward.

Works today

The status model

Four states today: received, accepted, rejected, archived. Environmental Health already wants about four more, mostly staff-side. Risk on both ends: bespoke statuses per MDA, or a generic Salesforce-style workflow engine. Keep it simple.

Open question

Timeline, CSV & SLA placeholders

Every status change and info exchange is logged to the case timeline. CSV export per program. The queue shows submission age and has SLA ideas written in — but they're just text, not enforced.

Works today

Working today

  • The forms → CMS handoff — submissions arrive with a program code and file themselves under the right program automatically.
  • Team-scoped queues — reverse-chronological list with submission date, age and applicant name; teams only see their own programs.
  • The request-more-information loop — question emailed out, typed answer lands back on the case page and timeline.
  • “View applicant page” — staff can see exactly what the citizen sees before changing a status.
  • Status changes, archive, CSV export, teams and access controls, API key admin, magic-link sign-in.

Gaps to close

  • Two different reference numbers — the Forms app email quotes an ID that isn't the CMS ID (fix in progress).
  • No entryway to the tracking page — it exists, but the link is never given out.
  • No CMS emails to applicants — approval changes nothing but a page nobody was told about; rejection is a dead end with no contact route.
  • Form data is hard to read on the case detail page — everything jumbled together.
  • No staff notifications or assignment — nobody is told a case arrived or whose it is.
  • Text-only info-request responses — no documents on the follow-up loop.

Open design questions

  • Who tells the applicant where their case went — the Forms app or the CMS — and what should that email say?
  • What are the real decision workflows per program (rolling vs batch, like summer camp's approve-all-at-once) — and what expectation copy follows: “hear back in X days” vs “this form typically responds in X”?
  • How do intermediate statuses work across MDAs (EHD wants ~4 more) without building either bespoke one-offs or a generic workflow engine?
  • Does a status page that mostly says “nothing has changed” create more demand than it absorbs? Push email on change + self-serve page likely both needed.
  • Staff-side notification and assignment model; better sign-in than magic links (SSO candidate).
